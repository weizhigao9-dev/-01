## 人工智能通识课项目说明
本知识库来自一份模拟的《人工智能通识课》课程笔记与实践项目说明，目标是让课程问答助手能够回答期末实践项目相关问题。
课程强调“理解原理、会用工具、能做作品、敢于反思”。实践项目的最终提交物包括项目报告、代码仓库、展示 PPT 和五分钟课堂演示。
项目评分采用多维度评价：任务完成度占 30%，独立思考占 30%，技术实现占 20%，结果展示与表达占 20%。其中，任务完成度关注是否完成指定功能；
独立思考关注是否有自己的问题分析、失败复盘和改进设想；技术实现关注代码结构、运行可复现性和工具使用的合理性；结果展示关注截图、测试表格、
演示逻辑和答辩表达。

## RAG 基本概念
RAG 是 Retrieval-Augmented Generation 的缩写，中文常译为检索增强生成。它的基本思想是把外部文档转化为可检索的知识库，
在用户提问时先检索相关资料，再把检索结果和问题一起交给大模型生成答案。与只依赖模型自身参数记忆的普通 LLM 相比，RAG 可以减少知识过期、
私有资料缺失和答案不可追溯的问题。一个典型 RAG 系统通常包含五个环节：文档加载、文本切分、向量化、相似度检索、带上下文生成。
文档加载负责读取 TXT、PDF、Word、网页等资料；文本切分负责把长文档切成适合检索的片段；向量化负责把片段转成语义向量；
向量数据库负责保存并快速查找相似片段；生成模型负责根据检索到的证据组织自然语言回答。

## 项目技术路线
本项目选择 LangChain 作为 RAG 编排框架，选择 Chroma 作为向量数据库，选择中文向量模型 bge-small-zh 或同类中文 embedding 模型进行文档向量化。
如果本地环境无法下载 embedding 模型，可以先使用离线关键词检索版完成课堂演示，再在联网环境中切换到完整 LangChain 流水线。
生成模型建议使用 DeepSeek V4 或 OpenAI 兼容接口，温度设置为 0 或 0.2，以减少随机性。项目的目录结构包括 data、src、outputs、docs 和 screenshots：
data 存放知识库原文；src 存放 RAG 主程序、评测脚本和 Streamlit 页面；outputs 存放问答结果；docs 存放报告和说明；screenshots 存放展示截图。

## 知识库内容与适用问题
知识库主要覆盖人工智能通识课的项目要求、RAG 原理、系统实现步骤、测试设计、失败案例分析和改进方向。它适合回答课程内部已有的信息，
例如“RAG 的两阶段流程是什么”“项目评分标准是什么”“为什么要设置文档内和文档外问题”“不同 chunk_size 对检索有什么影响”。
它不适合回答文档没有提供的信息，例如某位同学的真实学号、学校教务系统的提交截止时间、DeepSeek 最新价格、教师个人邮箱等。
当问题不在知识库中时，系统应该明确说明“知识库中没有找到依据”，而不是编造答案。

## 文本切分策略
文本切分是 RAG 效果的关键因素之一。chunk_size 过小，例如 200 字，优点是定位精确，缺点是上下文容易被切断，模型可能只看到半句话或半个论证。
chunk_size 中等，例如 500 字，通常能兼顾主题完整性和检索精度，是课程项目中较稳妥的默认值。chunk_size 过大，例如 1000 字，优点是上下文完整，
缺点是召回片段包含的信息太杂，可能降低相似度排序的区分度，并增加生成阶段的上下文长度。chunk_overlap 用于减少边界信息丢失，本项目默认使用
80 到 100 字重叠。实验结论是：课程笔记类文档适合 500 字左右的切块；概念定义类问答适合更小切块；长流程或案例分析适合稍大切块。

## 检索与回答策略
检索阶段采用 top_k=3，即每次返回最相关的三个文档片段。这样既能给生成模型足够证据，又不会把太多无关内容塞进提示词。
回答阶段使用固定提示词，要求模型只依据提供的资料回答。如果资料不足，必须说明无法从知识库确认。为了提高可追溯性，回答中应保留来源片段编号、
文档标题或页码。对于期末项目展示，最重要的不是让答案显得很长，而是让答案有依据、有边界、有对比。普通 LLM 容易给出通用但不可验证的答案；
RAG 的优势在于能引用课程文档中的具体要求和项目配置。

## 五个测试问题设计
测试集包含五个问题，其中三个问题在知识库中有明确答案，两个问题在知识库中没有答案。文档内问题包括：
一、RAG 的基本流程是什么；二、本课程实践项目评分标准是什么；三、chunk_size 设为 200、500、1000 时有什么差异。
文档外问题包括：一、任课教师下周三的办公室具体地点在哪里；二、某同学的真实学号是多少。这样设计的原因是，RAG 项目不能只展示会回答的问题，
还要证明系统能识别知识边界。一个可靠的知识库问答助手应该在资料不足时拒绝臆测。

## 普通 LLM 与 RAG 的差异
普通 LLM 不带检索时，只能依赖训练时学到的通用知识和提示词中的少量上下文。它回答 RAG 原理这类通用问题时通常表现不错，
但回答课程评分标准、项目目录结构、具体测试设计时可能出现笼统回答或幻觉。RAG 在回答文档内问题时会先检索课程笔记片段，
因而更容易给出“任务完成度 30%、独立思考 30%、技术实现 20%、结果展示 20%”这样的具体答案。面对文档外问题时，
RAG 如果提示词设置正确，会说明知识库没有依据；普通 LLM 则可能根据常识猜测。

## 失败案例一：检索不到
第一个失败案例是检索不到。原因可能是问题表述和文档表述差异太大，例如用户问“外挂资料库怎么接到模型上”，而文档只写“检索增强生成”。
改进方案包括增加同义词、使用更好的中文 embedding 模型、调整 chunk_size、加入 query rewrite，让系统先把口语化问题改写成更接近文档表达的检索词。
也可以在知识库中增加术语表，把 RAG、向量数据库、embedding、召回、top_k 等概念统一解释。

## 失败案例二：回答错误
第二个失败案例是检索到了相关片段，但生成模型仍然回答错误。常见原因包括提示词约束不足、多个片段内容冲突、模型过度发挥、引用片段太长导致重点不清。
改进方案是把提示词改成“只能依据资料回答”，给每个片段加来源编号，降低 temperature，并在回答后增加自检步骤：
判断答案中的每个关键事实是否能在检索片段中找到。如果找不到，就把该事实删除或标记为不确定。

## Web 界面设计
为了便于展示，本项目提供一个 Streamlit 页面。页面左侧可以设置 top_k、chunk_size 和是否显示来源片段；主区域提供问题输入框和回答展示区；
下方展示检索片段和相似度分数。课堂演示时，可以先提问一个文档内问题，展示 RAG 能找到依据；再提问一个文档外问题，展示系统会拒绝编造；
最后切换 chunk_size，说明参数变化对检索结果的影响。Web 页面不是 RAG 的核心，但能让项目更容易被老师和同学理解。

## 项目反思
这个项目的核心收获是理解大模型并不等于完整应用。真正可用的问答系统还需要文档治理、检索策略、提示词约束、评测集和失败复盘。
RAG 的优点是更新知识方便、答案可以追溯、适合课程资料、企业手册、个人笔记等私有场景。RAG 的局限是对原始文档质量非常敏感，
如果资料本身过时、分散、重复或矛盾，系统也会受影响。未来可以加入多格式文档解析、网页爬取、混合检索、重排序模型、用户反馈记录和自动评测脚本。

## 项目补充材料

### 课堂资料整理记录
本项目的知识库不是把所有文字简单堆在一起，而是先按照“课程要求、技术原理、系统实现、测试设计、失败分析、展示准备”的顺序整理。课程要求部分用于回答作业交付物、评分维度和展示形式；技术原理部分用于回答 RAG 与普通 LLM 的差异；系统实现部分用于解释 LangChain、Chroma、embedding、top_k 等工具配置；测试设计部分用于说明为什么要同时设置文档内问题和文档外问题。这样整理的好处是检索时主题更集中，模型拿到的上下文更容易直接支持答案。

### 文档清洗与切分准备
在真实项目中，原始文档往往包含标题、目录、表格、截图说明、重复页眉页脚和无关格式。进入向量库前，需要先做基础清洗：删除明显重复的页眉页脚，保留章节标题，把长表格转成简短说明，并统一中英文术语。例如“检索增强生成”“RAG”“Retrieval-Augmented Generation”在文档中应尽量同时出现，方便不同提问方式都能命中。切分时不宜只按固定字数硬切，而应优先按章节、自然段和句号切分，再用 chunk_size 控制片段长度。

### 评测问题记录
测试问题分为两组。第一组是文档内问题，包括“RAG 的基本流程是什么”“项目评分标准是什么”“chunk_size 参数有什么影响”。这些问题能检验系统是否找到知识库中的明确事实。第二组是文档外问题，包括“任课教师办公室地点在哪里”“某同学真实学号是多少”。这些问题用于检验系统是否会编造。一个合格的 RAG 项目不应该只展示能回答的问题，还要展示不知道时如何处理，因为拒绝无依据回答也是知识库问答系统的重要能力。

### 提示词模板设计
本项目的提示词采用保守策略：要求模型只依据参考资料回答，如果资料不足就说明知识库中没有找到依据。这个约束看似简单，但对减少幻觉很重要。提示词还可以加入三个细节：第一，要求答案尽量引用来源片段编号；第二，要求回答前先判断问题是否属于知识库范围；第三，要求回答后检查关键事实是否能在资料中找到。对于课程展示版本，不需要把提示词写得特别复杂，重点是让老师能看懂系统如何从“检索证据”走到“生成答案”。

### 参数实验补充
chunk_size=200 时，检索片段更短，适合概念定义类问题，例如“什么是 embedding”。缺点是如果答案需要上下文，片段可能不完整。chunk_size=500 时，通常能保留一个完整小节，适合课程笔记和项目说明，是本项目默认值。chunk_size=1000 时，上下文更充分，适合流程说明和案例复盘，但片段里可能混入多个主题，导致相似度排序不够精确。top_k 也需要配合调整：top_k 太小可能漏掉证据，top_k 太大则可能引入噪声。

### 系统边界说明
RAG 并不能保证所有答案都正确。它依赖三个前提：知识库里确实有答案，检索器能找到相关片段，生成模型愿意遵守提示词约束。如果课程资料本身过时、重复或互相矛盾，系统仍可能输出不稳定答案。因此，报告中需要把系统边界写清楚：本项目适合回答课程笔记、项目要求和实验记录相关问题，不适合回答实时通知、个人隐私、教师行程、未收录文件和需要联网查询的问题。明确边界比假装无所不知更专业。

### 展示流程脚本
课堂展示可以控制在五分钟内。第一分钟说明任务目标：做一个能回答自有文档问题的 RAG 助手。第二分钟展示系统架构：文档加载、切分、向量化、检索和生成。第三分钟演示文档内问题，例如评分标准，让系统给出具体比例。第四分钟演示文档外问题，例如教师办公室地点，让系统说明没有依据。第五分钟总结对比结果和失败改进，包括 chunk_size、top_k、提示词约束和多格式文档解析。这样展示逻辑完整，也能体现独立思考。

### 后续优化设想
如果继续完善，可以从四个方向升级。第一，支持 PDF、Word、网页等多格式文档，让知识库来源更丰富。第二，加入混合检索，把关键词检索和向量检索结合起来，减少术语不一致造成的漏检。第三，加入 rerank 重排序模型，对初步召回片段重新排序，提高证据质量。第四，加入用户反馈记录，把“回答有用/无用”和失败问题保存下来，作为下一轮优化依据。最终目标不是做一个一次性演示脚本，而是形成可持续维护的个人知识库问答工具。

### 依赖环境说明
完整版本建议使用 Python 3.10 或以上版本，安装 langchain、langchain-community、langchain-openai、langchain-chroma、langchain-huggingface、chromadb、sentence-transformers、streamlit 和 python-dotenv。DeepSeek 接口采用 OpenAI 兼容调用方式，环境变量名可以设置为 DEEPSEEK_API_KEY。由于不同电脑的网络环境不同，embedding 模型第一次下载可能较慢，因此项目保留 offline_demo.py 作为兜底演示。离线脚本不等于完整 RAG，但它能展示检索、对比测试、拒绝编造和结果保存的核心流程。

### 数据安全与隐私提醒
如果以后把个人简历、聊天记录或课程作业放入知识库，需要注意隐私边界。知识库文件中不应包含身份证号、真实电话、家庭住址、账号密码、同学隐私或未经允许公开的教师资料。RAG 系统虽然是本地项目，但如果调用云端大模型，检索片段会作为提示词发给模型服务商。因此，正式使用前要先做脱敏处理，把敏感信息替换为占位符。对于课堂项目，可以优先使用课程笔记、公开资料和自己编写的项目说明，既能满足测试要求，也更安全。

### 评分要求对照
从评分角度看，本项目可以对应四类证据。任务完成度方面，知识库文档、RAG 主程序、五个测试问题和结果表格证明任务已经完成。独立思考方面，失败案例、参数实验和系统边界说明体现了不只是在照抄代码。技术实现方面，目录结构、requirements、LangChain 流水线、Streamlit 页面和离线演示脚本说明项目具有可复现性。结果展示方面，报告、PPT、截图和问答对比表可以支撑课堂五分钟演示。提交前应检查这些证据是否都在仓库中。

### 常见问题准备
答辩时可能会被问到几个问题。第一，为什么不用微调？回答是本项目知识会更新，文档量不大，RAG 更轻量。第二，为什么要设置文档外问题？回答是为了测试系统是否能识别知识边界。第三，为什么普通 LLM 也能回答部分问题？回答是通用概念它可能知道，但课程评分标准和项目细节需要私有资料支撑。第四，为什么使用 chunk_size=500？回答是它在精确性和上下文完整性之间比较平衡。第五，项目最大的不足是什么？回答是还没有接入真实多格式解析和自动化评测。

### 运行记录说明
本项目运行后会在 outputs 目录生成 evaluation_results.csv 和 evaluation_results.md。CSV 文件适合放入报告或作为测试证明，Markdown 文件适合在 GitHub 上直接查看。测试结果不是只看答案是否流畅，而是看答案是否符合资料、是否能处理未知问题、是否能说明普通 LLM 与 RAG 的差异。正式运行完整版本时，建议每次修改知识库后重新生成向量库，并记录 chunk_size、top_k、模型名称、温度参数和测试日期，这样报告中的实验结论更可信。

### 个人学习收获
通过这个项目，我最大的收获是理解了大模型应用的关键不只是“调用一个 API”。一个看起来简单的知识库问答助手，背后需要考虑文档来源、文本清洗、切分粒度、检索质量、提示词约束、结果评测和用户交互。RAG 的优势在于把大模型的语言表达能力和外部文档的事实依据结合起来，但它也要求开发者对资料负责。以后无论是做课程助手、个人笔记助手还是企业知识库，都需要先把知识组织好，再让模型回答。
